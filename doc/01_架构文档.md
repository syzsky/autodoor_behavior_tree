# AutoDoor 行为树系统 — 架构文档

## 1 项目概述

### 1.1 项目定位

AutoDoor 行为树系统是一个独立的可视化行为树编辑与执行框架，面向 Windows 平台的自动化场景（客户端辅助、RPA 流程等）。系统提供完整的图形化编辑器、多种检测/动作节点、脚本录制、OCR 识别、DD 虚拟键盘等能力，并支持标准版与 DD 版两种打包模式。

### 1.2 核心特性

| 特性 | 说明 |
|------|------|
| 可视化编辑器 | 基于 CustomTkinter 的节点式编辑器，支持拖拽、连线、缩放、框选 |
| 行为树引擎 | 独立线程执行，支持启动/暂停/停止/恢复 |
| 21 种内置节点 | 5 种复合节点 + 6 种条件节点 + 10 种动作节点 |
| 黑板系统 | 观察者模式的数据共享机制，节点间解耦通信 |
| 序列化 | JSON/YAML/TXT 多格式持久化，版本化数据结构 |
| 撤销/重做 | Command 模式，支持 100 步历史 |
| DD 虚拟键盘 | DD级硬件模拟输入，绕过大多数输入检测 |
| OCR 集成 | 内嵌 RapidOCR + ONNX Runtime，支持中英文识别 |
| 文本处理 | 文本提取与输入功能，支持 OCR 提取、预设文本、文件输入 |
| 脚本录制 | TXT 脚本录制与回放 |
| 双版打包 | 标准版 (PyAutoGUI) 与DD版 (DD64.dll) |
| 放大镜工具 | 区域/位置选择时显示放大镜，精确定位 |
| 随机范围 | 时长参数支持随机变化，模拟人类操作 |
| 坐标偏移 | 条件节点支持坐标偏移，可视化测量工具 |
| 子树引用 | SubtreeNode 引用外部行为树项目，支持循环检测、黑板隔离、预览展开 |
| 多树并行 | MultiTreeManager 管理多个独立行为树实例，支持共享黑板 |
| 窗口绑定 | 支持绑定目标窗口实现后台运行，自动坐标转换与智能窗口切换 |

### 1.3 技术栈

```
Python 3.10+
├── GUI 框架:    CustomTkinter + Tkinter Canvas
├── 图像处理:    Pillow, OpenCV, imagehash
├── OCR 引擎:    RapidOCR + ONNX Runtime（支持中英文识别）
├── 输入模拟:    PyAutoGUI (标准版) / DD64.dll (DD版)
├── 音频播放:    pygame.mixer
├── 打包工具:    PyInstaller
└── CI/CD:       GitHub Actions
```

## 3 核心组件设计

### 3.1 节点模型 (bt_core/nodes.py)

节点模型是整个系统的核心抽象，采用经典的 Composite 模式设计：

```
Node (ABC)
├── CompositeNode (ABC)
│   ├── SequenceNode     — 顺序执行，全部成功才成功（支持 continue_on_failure、子节点间隔）
│   ├── SelectorNode     — 选择执行，任一成功即成功（支持子节点间隔）
│   ├── ParallelNode     — 并行执行，策略决定成功条件
│   ├── RandomNode       — 随机执行子节点，支持成功策略和完全随机模式
│   └── StartNode        — 根节点，顺序执行子节点，失败后继续执行后续子节点
│   └── SubtreeNode      — 子树引用节点，加载外部行为树项目执行
├── ConditionNode (ABC)
│   ├── _children_running: bool  — 子节点执行状态追踪标志
│   ├── offset: Tuple[int, int]  — 坐标偏移量（相对偏移）
│   ├── _validate_region(region) — 验证区域配置是否有效
│   ├── _get_region_image(context) — 获取区域图像（支持固定区域/动态区域/回退区域）
│   ├── _resolve_dynamic_region(context) — 根据黑板锚点+偏移量计算动态区域
│   ├── _get_fallback_region(context) — 无region时使用上次检测位置附近区域（V1.5.2a 新增）
│   ├── _save_position(context, position) — 保存位置到黑板（自动拆分x/y）
│   ├── _log_condition_result(success, reason, extra_info) — 记录条件检测结果日志
│   ├── _STABLE_THRESHOLD_LOW/HIGH — 智能跳过阈值（V1.5.2a 新增）
│   ├── OCRConditionNode     — 文字检测（支持识别方向参数）
│   ├── ImageConditionNode   — 图像模板匹配
│   ├── ColorConditionNode   — 颜色检测
│   ├── NumberConditionNode  — 数字识别比较（支持识别方向参数）
│   ├── VariableConditionNode— 变量值判断（支持常量/变量比较、存在性判断）
│   └── TextExtractNode      — 文本提取（OCR识别后提取文本到黑板）
└── ActionNode (ABC)
    ├── _children_running: bool  — 子节点执行状态追踪标志
    ├── KeyPressNode     — 键盘按键
    ├── MouseClickNode   — 鼠标点击
    ├── MouseMoveNode    — 鼠标移动（支持起点到终点移动、增量移动、拖拽）
    ├── MouseScrollNode  — 鼠标滚动
    ├── DelayNode        — 延时等待
    ├── SetVariableNode  — 设置变量（支持常量赋值/变量赋值、递增、删除）
    ├── ScriptNode       — 执行脚本
    ├── CodeNode         — 执行代码
    ├── AlarmNode        — 播放报警
    ├── TextInputNode    — 文本输入（支持提取值/预设/文件三种输入模式）
    ├── StartTreeNode    — 启动其他行为树
    └── StopTreeNode     — 停止当前或其他行为树
```

**节点核心接口：**

| 方法 | 说明 |
|------|------|
| `tick(context)` | 执行节点逻辑，返回 NodeStatus |
| `reset(reset_counters)` | 重置节点状态，`reset_counters` 控制是否重置计数器 |
| `abort(context)` | 中止节点执行 |
| `add_child(child)` | 添加子节点 |
| `remove_child(child)` | 移除子节点 |
| `to_dict()` | 序列化为字典 |
| `from_dict(data)` | 从字典反序列化 |
| `_execute_children(context)` | 执行子节点（通用实现，位于 Node 基类） |

**子节点执行逻辑统一：**

`_execute_children()` 方法已统一到 `Node` 基类，`ConditionNode` 和 `ActionNode` 不再各自实现：

```python
class Node:
    def __init__(self, ...):
        ...
        self._child_index = 0
        self._children_running = False
    
    def _execute_children(self, context) -> NodeStatus:
        """执行子节点（通用实现）"""
        while self._child_index < len(self.children):
            child = self.children[self._child_index]
            status = child.tick(context)
            if status == NodeStatus.RUNNING:
                self._children_running = True
                return NodeStatus.RUNNING
            if status != NodeStatus.SUCCESS:
                self._child_index = 0
                self._children_running = False
                return status
            self._child_index += 1
        self._child_index = 0
        self._children_running = False
        return NodeStatus.SUCCESS
``` |

**节点保护机制：**

节点支持保护属性 `_is_protected`，防止被误删除：

| 节点类型 | 默认保护状态 | 说明 |
|----------|-------------|------|
| StartNode | 受保护 | 根节点不可删除，确保行为树结构完整 |
| 其他节点 | 不受保护 | 可自由删除 |

```python
# 检查节点是否受保护
if not node._is_protected:
    # 允许删除操作
    pass
```

**节点日志输出机制：**

所有节点在执行成功或失败时都会通过 LogManager 输出运行日志：

| 日志类型 | 触发时机 | 输出内容 |
|----------|----------|----------|
| 成功日志 | 节点返回 SUCCESS | 节点类型、节点名称 |
| 失败日志 | 节点返回 FAILURE | 节点类型、节点名称、失败原因 |
| 中止日志 | 节点被 abort() | 节点类型、节点名称 |

**失败原因输出规范：**

| 节点类型 | 失败原因示例 |
|----------|-------------|
| SequenceNode | "子节点 [节点名] 执行失败" |
| SelectorNode | "所有子节点执行失败" |
| ParallelNode | "成功 2/5 个子节点" |
| RandomNode | "成功 2/5 个子节点" 或 "所有子节点都执行失败" |
| StartNode | "部分子节点执行失败" |
| OCRConditionNode | "未找到关键词: '登录'，识别到的文本: 注册 取消" |
| ImageConditionNode | "未找到匹配的图像，最高置信度: 45.23%，阈值: 80.00%" |
| ColorConditionNode | "未找到匹配的颜色 #FF0000，容差: 30" |
| NumberConditionNode | "未识别到数字，识别到的文本: ABC DEF" |
| VariableConditionNode | "变量 'count' 不存在" 或 "变量 'count' 值为 5，不满足 > 10" 或 "比较变量 'source' 不存在或值为 None" |

**成功日志简化原则：**

成功日志仅输出节点类型和名称，不包含额外信息，保持简洁：

```python
LogManager.instance().log_success(
    node_type="鼠标移动节点",
    node_name="移动到按钮位置"
)
```

**节点状态管理机制：**

节点在重试（retry）和重复执行（repeat）时需要正确管理状态：

| 方法 | 说明 |
|------|------|
| `_reset_for_retry()` | 重试时重置状态，设置 `status = RUNNING`，保留重试计数器 |
| `_reset_for_repeat()` | 重复执行时重置状态，设置 `status = RUNNING`，保留重复计数器 |
| `_execute_with_decorators()` | 装饰器模式处理 retry/repeat/timeout |

**重要：** `_reset_for_retry()` 和 `_reset_for_repeat()` 忺须将 `status` 设置为 `RUNNING`，而非 `SUCCESS`。这是因为在重试/重复期间，节点仍处于执行状态，需要返回 `RUNNING` 让引擎继续 tick。

**`_reset_for_retry()` 必须重置的字段（V1.3.2b 修复）：**

| 字段 | 说明 | 重置值 |
|------|------|--------|
| `status` | 节点状态 | `RUNNING` |
| `_tick_count` | tick 计数 | `0` |
| `_start_time` | 开始时间 | `None` |
| `_child_index` | 子节点执行索引（基类） | `0` |
| `_children_running` | 子节点运行标志 | `False` |
| `current_index` | 组合节点子节点索引（CompositeNode） | `0` |
| `_last_child_finish_time` | 子节点完成时间（SequenceNode） | `None` |
| `_last_child_time` | 子节点时间（SelectorNode） | `0` |
| 所有子节点 | 递归调用 `child.reset()` | — |

**修复背景：** 在 V1.3.2b 之前，`_reset_for_retry()` 未重置 `current_index`、`_child_index`、`_children_running` 等字段，导致组合节点重试时状态不一致，可能出现：
- ConditionNode 跳过条件检查直接执行子节点
- SequenceNode 从错误的子节点索引继续执行
- 状态灯显示与实际执行结果不一致

**子节点执行状态追踪机制：**

ActionNode 和 ConditionNode 在执行子节点时需要追踪子节点的执行状态，避免在子节点运行期间重复执行动作/条件检测：

| 属性 | 说明 |
|------|------|
| `_children_running` | 布尔标志，标记子节点是否正在执行 |

**状态追踪流程：**

```
ActionNode/ConditionNode 执行流程:
┌─────────────────────────────────────────────────────────────────────┐
│  1. 首次 tick:                                                       │
│     - 执行动作/条件检测                                               │
│     - 若成功且有子节点:                                               │
│       - 通知 GUI 节点状态为 SUCCESS                                   │
│       - 设置 _children_running = True                                │
│       - 开始执行子节点                                                │
│                                                                      │
│  2. 后续 tick:                                                       │
│     - 检查 _children_running 标志                                    │
│     - 若为 True，跳过动作/条件检测，继续执行子节点                      │
│     - 子节点执行完成后，重置 _children_running = False                │
└─────────────────────────────────────────────────────────────────────┘
```

**GUI 状态通知机制：**

当 ActionNode 或 ConditionNode 成功执行后需要执行子节点时，必须先通知 GUI 节点状态为 SUCCESS，确保界面正确显示节点状态：

```python
# 在执行子节点前通知 SUCCESS 状态
if status == NodeStatus.SUCCESS and self.children:
    context.notify_node_status(self.node_id, "success")
    self._children_running = True
    return self._execute_children(context)
```

**reset 方法签名：**

所有节点的 `reset()` 方法应接受 `reset_counters: bool = True` 参数：

```python
def reset(self, reset_counters: bool = True) -> None:
    """
    重置节点状态
    
    Args:
        reset_counters: 是否重置重试/重复计数器，默认 True
    """
    if reset_counters:
        self._retry_count = 0
        self._repeat_count = 0
    self.status = NodeStatus.SUCCESS
    self._tick_count = 0
    self._start_time = None
```

### 3.2 执行引擎 (bt_core/engine.py)

BehaviorTreeEngine 负责行为树的生命周期管理：

- **执行模式**：独立守护线程循环 tick
- **Tick 间隔**：默认 10ms，可配置
- **状态回调**：`on_status_change` / `on_node_status` 通知外部
- **线程安全**：`threading.Lock` 保护启动/停止操作

def _run_loop(self):
    while not self._stop_event.is_set():
        if not self._pause_event.wait(0.1):

        self._tick()
        self._stop_event.wait(self._tick_interval)

### 3.3.1 命名空间黑板 (NamespacedBlackboard)

NamespacedBlackboard 是 Blackboard 的代理实现，用于子树场景下的数据隔离：

| 黑板模式 | 说明 | 适用场景 |
|----------|------|----------|
| inherit | 共享父级黑板引用 | 默认模式，子树直接读写父树变量 |
| isolated | 独立黑板实例 | 子树拥有完全隔离的数据空间 |
| namespaced | 命名空间前缀隔离 | 变量自动加命名空间前缀，避免冲突 |

**核心接口：**

| 方法 | 说明 |
|------|------|
| `_wrap_key(key)` | 为键添加命名空间前缀 |
| `get/set/increment/delete` | 代理到父级黑板，使用带前缀的键 |
| `get_all_keys()` | 仅返回当前命名空间的键 |
| `subscribe/unsubscribe` | 代理订阅/取消订阅 |

**可配置变量名：**

条件节点支持 `position_key` 参数，允许自定义位置保存的黑板变量名，支持多检测点场景。`position_key` 默认为空，为空时使用全局默认值（由 `ConfigManager` 管理，默认为 `last_detection_position`）。

**坐标自动拆分：**

`_save_position` 方法在保存位置到黑板时，自动将坐标拆分为 x/y 分量写入 `last_detection_x` 和 `last_detection_y`。这两个变量始终对应全局最近检测位置，与自定义 `position_key` 无关。典型用法：通过 VariableConditionNode 对 y 坐标进行数值比较，判断检测目标在屏幕的上/下方位。

**动态区域检测：**

条件节点支持两种区域选择方式，通过 `region_mode` 配置切换：

| 模式 | 内部值 | 说明 |
|------|--------|------|
| 固定区域检测 | `"fixed"` | 默认模式，使用 `region` 字段指定的固定坐标区域 |
| 动态区域检测 | `"dynamic"` | 根据黑板锚点位置 + 偏移量动态计算检测区域 |

动态区域检测的配置参数：

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `region_use_last_pos` | 使用最近检测点作为锚点 | `True` |
| `region_anchor` | 自定义锚点变量名（`region_use_last_pos` 为 False 时生效） | `""`（空时回退到 `last_detection_position`） |
| `region_offset` | 区域偏移量 `[x1, y1, x2, y2]` | `[-50, -50, 50, 50]` |

动态区域计算公式：`region = (anchor_x + offset_x1, anchor_y + offset_y1, anchor_x + offset_x2, anchor_y + offset_y2)`

当锚点变量不存在时，`_resolve_dynamic_region` 输出警告日志并返回 `None`，`_get_region_image` 降级为全屏截图。

#### 单树快捷键架构

单树快捷键在全局快捷键基础上扩展，为每个行为树 Tab 绑定独立的启动/停止快捷键：

- **数据流**：SettingsTab → SettingsManager → Editor._register_tab_shortcuts() → GlobalHotkeyManager
- **配置存储**：`shortcuts.tab_shortcuts` 数组，每项包含 `hotkey`（快捷键名）和 `tab_name`（行为树名称，可选）
- **Toggle 逻辑**：同一快捷键同时注册启动和停止，`_toggle_tab()` 方法根据当前运行状态决定执行启动还是停止
- **组合键匹配**：`_check_hotkey()` 中，有修饰键按下时只匹配组合键（如 `shift+f9`），无修饰键时只匹配单键（如 `f9`），避免组合键误触发单键

#### 组合键架构

GlobalHotkeyManager 支持组合键识别：

- **修饰键追踪**：`_pressed_modifiers` 集合实时追踪当前按下的修饰键（Ctrl/Alt/Shift/Cmd）
- **修饰键归一化**：`_normalize_modifier()` 将 pynput 返回的变体名（如 shiftleft/shiftright）统一为标准名（shift）
- **组合键构建**：非修饰键按下时，将当前修饰键集合与非修饰键组合，排序后形成 `ctrl+f10` 格式的组合键名
- **冲突避免**：有修饰键时只匹配组合键候选，无修饰键时只匹配单键候选，确保 F9 和 Shift+F9 不会互相干扰
- **捕获安全**：SettingsTab 捕获快捷键时暂停 GlobalHotkeyManager，捕获完成后恢复，避免修改快捷键时误触发

def _schedule_drag_redraw(self):
    """节流式拖拽重绘（16ms/60fps）"""

    else:
        self._drag_pending_redraw = True

## 5 基础设施层

### 5.1 输入控制器

系统采用多引擎、键盘/鼠标分离的输入架构，通过配置文件 `input.keyboard_method` 和 `input.mouse_method` 独立选择键盘和鼠标引擎：

| 引擎 | 实现文件 | 输入层级 | 特点 |
|------|----------|----------|------|
| PyAutoGUI | input_controller.py | APPLICATION | 跨平台兼容，易被检测 |
| DD | dd_input.py | DRIVER | 内核驱动级模拟，绕过检测 |
| IbInputSimulator | ib_input.py | DRIVER | Hook+SendInput，支持6种驱动模式 |
| Background | bg_input.py | BACKGROUND | PostMessage后台操作，窗口不可见时可用 |

**InputLevel 枚举：**

```
InputLevel
├── APPLICATION   # 应用层（PyAutoGUI）
├── DRIVER        # 驱动层（DD、IbInputSimulator）
└── BACKGROUND    # 后台层（PostMessage）
```

> 注：KERNEL 级别随 Interception 删除已移除。

**基类层次结构：**

键盘和鼠标控制器分别抽象，通过组合模式构成完整输入控制器：

```python
class BaseKeyboardController(ABC):
    """键盘控制器基类"""
    def key_press(self, key, action="press", duration=0)
    def key_down(self, key)
    def key_up(self, key)
    @classmethod
    def is_simulating(cls) -> bool
    @classmethod
    def release_all(cls)

class BaseMouseController(ABC):
    """鼠标控制器基类"""
    def mouse_click(self, button="left", position=None, action="press", duration=0)
    def mouse_down(self, button="left")
    def mouse_up(self, button="left")
    def mouse_move(self, position, relative=False)
    def mouse_scroll(self, amount, position=None)
    def move_to(self, x, y)
    def get_position(self) -> Tuple[int, int]

class BaseInputController(BaseKeyboardController, BaseMouseController):
    """组合基类：同时实现键盘和鼠标接口"""
    pass
```

**InputControllerManager（input_manager.py）：**

管理双引擎缓存，键盘和鼠标引擎独立选择：

```python
class InputControllerManager:
    _keyboard_engines: Dict[str, BaseKeyboardController]  # 键盘引擎缓存
    _mouse_engines: Dict[str, BaseMouseController]        # 鼠标引擎缓存

    def get_keyboard_engine(self, method: str) -> BaseKeyboardController
    def get_mouse_engine(self, method: str) -> BaseMouseController
```

配置示例：
```ini
[input]
keyboard_method = ib        # 键盘使用 IbInputSimulator
mouse_method = pyautogui    # 鼠标使用 PyAutoGUI
```

**IbInputSimulatorInput（ib_input.py）：**

基于 Hook + SendInput 的驱动层输入引擎，同时支持键盘和鼠标操作：

- **初始化**: `IbSendInit(send_type, mode, args)` — send_type 为整数，对应6种驱动模式：
  - `0` = AnyDriver（自动选择可用驱动）
  - `1` = SendInput（移除 injected 标志，使输入表现为硬件输入）
  - `2` = Logitech（罗技驱动）
  - `3` = Razer（雷蛇驱动）
  - `5` = MCII（需传入目标 PID: `IbSendInit(5, 0, pid)`）
  - `6` = LogitechGHubNew（罗技 G Hub 新版驱动）
- **Hook 机制**: `IbSendInputHook(1)` 开启 Hook，`IbSendInputHook(0)` 关闭 Hook
  - Hook 拦截 SendInput API 调用，将其重定向到对应驱动
- **鼠标操作**:
  - 移动: `SetCursorPos` + `SendInput(MOVE|ABSOLUTE)` 设置绝对位置
  - 点击: `SendInput(button flags)` 发送按键事件
- **MCII 模式**: 需通过 `IbSendInit(5, 0, pid)` 传入目标进程 PID
- **SendInput 模式**: 移除 injected 标志，使输入表现为硬件输入

**BackgroundInputController（bg_input.py）：**

基于 PostMessage 的后台输入控制器，支持窗口不可见时操作：

- 键盘: `PostMessage(hwnd, WM_KEYDOWN/WM_KEYUP, vk_code, lparam)`
- 鼠标移动: 支持绝对坐标和相对坐标两种模式
  - 绝对: `PostMessage(hwnd, WM_MOUSEMOVE, 0, MAKELPARAM(x, y))`
  - 相对: `PostMessage(hwnd, WM_MOUSEMOVE, 0, MAKELPARAM(delta_x, delta_y))`
- 鼠标点击: `PostMessage(hwnd, WM_LBUTTONDOWN/WM_LBUTTONUP, ...)`
- 鼠标滚轮: `PostMessage(hwnd, WM_MOUSEWHEEL, WHEEL_DELTA, MAKELPARAM(x, y))`
- 后台模式下键盘和鼠标引擎各自独立接收 hwnd 参数

**DDVirtualInput（dd_input.py）：**

内核驱动级输入，需要管理员权限和驱动安装：

```
1. 加载 DD64.dll → DD_btn(0) 验证
2. 按键转换: key_name → VK_CODE_MAP → DD_todc() → DD_code
3. 执行: DD_key(dd_code, 1/2) 按下/抬起
4. 鼠标: DD_mov/DD_movR/DD_btn/DD_whl
```

**DD鼠标滚轮实现细节（v2.0增强版）：**

- **API限制**: `DD_whl()` 只接受方向码（1=向上/向前，2=向下/向后），不支持连续滚动量
- **解决方案**: 循环调用 `DD_whl(1或2)` 实现大距离滚动
- **智能算法**: 动态计算间隔 `interval = 1秒 / 总次数`，确保单次滚动≤1秒
- **三重保护机制**:
  1. **启动前检查**: 检测到停止标志则跳过（防止clicks循环重复启动）
  2. **超时保护**: 单次滚动最长1秒，防止长时间占用
  3. **中断响应**: 每次循环检查F12停止请求，可立即中断
- **全局停止**: F12触发 `DDVirtualInput.request_stop()` → 设置类变量 `_stop_requested=True`
- **Normal版本兼容**: `scroll.py` 的 clicks 循环增加 `is_running` 检查，两个版本都受益

### 5.2 OCR 管理器

OCRManager 封装 RapidOCR + ONNX Runtime，单例模式：

- `recognize(image, keywords, language, preprocess_mode, region)` — 关键词检测，返回 (found, position, all_text)
- `recognize_number(image, language, preprocess_mode, extract_mode, extract_pattern, min_confidence)` — 数字识别，返回 (found, value, all_text)
- `recognize_number_with_position(...)` — 数字识别（带位置），返回 (found, value, all_text, position)
- `get_all_text(image, language, preprocess_mode)` — 全文识别
- `clear_cache()` — 清空结果缓存

**结果缓存机制：**

为提升性能，OCRManager 内置结果缓存：

| 配置 | 值 | 说明 |
|------|-----|------|
| DEFAULT_CACHE_TTL | 1.0 秒 | 缓存过期时间 |
| MAX_CACHE_SIZE | 200 条 | 最大缓存条目数（V1.5.2a 从50提升至200） |

缓存键基于图像哈希和参数哈希生成，相同图像+参数的重复识别直接返回缓存结果。

**OCR缓存命中率统计（V1.5.2a 新增）：**

OCRManager 新增 `_cache_hits` / `_cache_misses` 计数器，支持通过 `get_cache_stats()` 获取缓存命中率统计，用于性能监控和调优。

**图像预处理模式：**

> 所有条件节点（OCR文字检测、数字比较、文本提取）共享统一的预处理模式映射，定义在 `bt_nodes/conditions/common.py`。

| 显示名称 | 内部值 | 说明 | 适用场景 |
|------|------|------|---------|
| 默认 | normal | 灰度→对比度增强→锐化→二值化 | 标准文字识别 |
| 复杂色彩 | game | 放大→中值滤波→灰度→对比度增强→锐化→二值化 | 游戏界面、小字体 |
| 自适应 | adaptive | CLAHE对比度增强→自适应二值化 | 低对比度、渐变背景 |
| 自动调优 | auto | 根据图像特征自动选择最佳预处理参数 | 所有场景，推荐使用 |

> 向后兼容：旧值"普通文本"映射到 normal，"艺术字"映射到 game。

**数字提取模式：**

| 模式 | 说明 |
|------|------|
| 无规则 | 提取第一个数字 |
| x/y | 提取分数形式的分子（如 "3/5" 提取 3） |
| 自定义 | 使用 `*` 通配符定义提取模式（如 "HP:*" 提取 "HP:100" 中的 100） |

1. 基于 ONNX Runtime，无需安装 Tesseract

    """获取虚拟桌面边界（支持多显示器）"""
    try:
        import screeninfo
        monitors = screeninfo.get_monitors()
        if monitors:
            min_x = min(monitor.x for monitor in monitors)
            min_y = min(monitor.y for monitor in monitors)
            max_x = max(monitor.x + monitor.width for monitor in monitors)
            max_y = max(monitor.y + monitor.height for monitor in monitors)
            return (min_x, min_y, max_x, max_y)
    except ImportError:
        pass
    
    # Windows API 备用方案
    try:
        import ctypes
        user32 = ctypes.windll.user32
        virtual_left = user32.GetSystemMetrics(76)   # SM_XVIRTUALSCREEN
        virtual_top = user32.GetSystemMetrics(77)    # SM_YVIRTUALSCREEN
        virtual_width = user32.GetSystemMetrics(78)  # SM_CXVIRTUALSCREEN
        virtual_height = user32.GetSystemMetrics(79) # SM_CYVIRTUALSCREEN
        return (virtual_left, virtual_top, 
                virtual_left + virtual_width, 
                virtual_top + virtual_height)
    except Exception:
        pass
    
    # 最终回退

### 5.8 随机范围辅助工具 (bt_utils/helpers.py)

提供随机值计算工具函数，用于模拟更自然的人类操作：

**核心函数：**

```python
def get_random_value(base_value: Union[int, float], random_range: Union[int, float] = 0, 
                     min_value: Union[int, float, None] = None) -> Union[int, float]:
    """
    计算带随机范围的值
    
    Args:
        base_value: 基础值（支持整数和浮点数）
        random_range: 随机范围（±）
        min_value: 最小值限制（可选）
    
    Returns:
        计算后的随机值
    """
    if random_range <= 0:
        return base_value
    
    min_val = base_value - random_range
    max_val = base_value + random_range
    
    if min_value is not None:
        min_val = max(min_val, min_value)
    
    if isinstance(base_value, float):
        return random.uniform(min_val, max_val)
    return random.randint(int(min_val), int(max_val))


def get_random_duration(base_ms: int, random_ms: int = 0) -> int:
    """
    计算随机延时时长（毫秒）
    
    Args:
        base_ms: 基础时长（毫秒）
        random_ms: 随机范围（±毫秒）
    
    Returns:
        计算后的延时时长（毫秒）
    """
    return get_random_value(base_ms, random_ms, min_value=0)


def get_random_interval(base_ms: int, random_ms: int = 0) -> float:
    """
    计算随机间隔时长（秒）
    
    Args:
        base_ms: 基础时长（毫秒）
        random_ms: 随机范围（±毫秒）
    
    Returns:
        计算后的间隔时长（秒）
    """
    ms = get_random_value(base_ms, random_ms, min_value=0)
    return ms / 1000.0
```

### 5.8 UI 更新分发器 (bt_utils/ui_dispatcher.py)

UIUpdateDispatcher 是解决 tkinter 线程安全问题的核心组件，采用事件驱动 + 定时轮询混合模式：

**设计背景：**

tkinter 的 GUI 操作必须在主线程执行，但行为树引擎在独立线程中运行。直接从后台线程调用 `update()` 或 `event_generate()` 可能导致界面假死或更新延迟。

**核心机制：**

| 机制 | 说明 |
|------|------|
| 线程安全队列 | `Queue` 存储待处理的 UI 更新任务 |
| 定时轮询 | 自适应间隔：运行时10ms，空闲时100ms（V1.5.2a 从固定30ms优化） |
| 节点状态缓存 | 合并相同节点的重复状态更新 |
| 后台轮询守护 | 当 `_widget` 为 None 时，后台线程继续尝试调度 |

**数据流：**

```
后台线程 (Engine)                     主线程 (GUI)
       │                                   │
       │  dispatch_node_status()           │
       │  ┌──────────────────────┐         │
       │  │ Queue.put(task)      │         │
       │  └──────────────────────┘         │
       │                                   │
       │  同时：定时轮询检查队列            │
       │  ┌──────────────────────┐         │
       │  │ 每30ms检查队列        │────────▶│
       │  │ 有任务则处理          │         │
       │  └──────────────────────┘         │
       │                                   │  _process_updates()
       │                                   │  ┌──────────────────────┐
       │                                   │  │ 从队列取出任务        │
       │                                   │  │ 执行回调更新UI        │
       │                                   │  └──────────────────────┘
```

**关键方法：**

| 方法 | 说明 |
|------|------|
| `attach(widget)` | 绑定到 GUI 组件 |
| `dispatch_node_status()` | 分发节点状态更新 |
| `dispatch_log_flush()` | 分发日志刷新请求 |
| `start_polling()` | 启动定时轮询 |
| `stop_polling()` | 停止定时轮询 |
| `process_pending()` | 立即处理待处理任务 |

**⚠️ 防止假卡死的重要注意事项：**

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 启动时假卡死 | 轮询未在编辑器初始化时启动 | 在 `_init_ui_dispatcher()` 中立即调用 `start_polling()` |
| 手动停止后再次运行异常 | 停止时调用 `stop_polling()` 导致轮询停止 | **不要在停止时调用 `stop_polling()`**，让轮询持续运行 |
| 轮询永久停止 | `_widget` 为 None 时轮询直接返回 | 使用后台线程守护，当 `_widget` 可用时切换回主线程轮询 |
| `after(0, ...)` 不可靠 | 从后台线程调用可能不生效 | 完全依赖定时轮询机制，不使用 `after(0, ...)` |

**正确的初始化流程：**

```python
def _init_ui_dispatcher(self):
    from bt_utils.ui_dispatcher import UIUpdateDispatcher
    self._dispatcher = UIUpdateDispatcher.get_instance()
    self._dispatcher.attach(self)
    self._dispatcher.start_polling()  # 必须在初始化时启动，而不是运行时

def _start_ui_polling(self):
    pass  # 已在初始化时启动，此处为空

def _stop_ui_polling(self):
    pass  # 不要停止轮询，让它持续运行
```

**轮询守护机制：**

```python
def _poll(self):
    if not self._polling_active:
        return
    
    widget = self._widget
    
    try:
        if widget is not None:
            pending = self._task_queue.qsize()
            if pending > 0:
                self._process_updates()
    except Exception:
        pass
    
    if self._polling_active:
        if self._widget is not None:
            self._widget.after(self._polling_interval_ms, self._poll)
        else:
            # 后台线程守护，等待 _widget 可用
            threading.Timer(self._polling_interval_ms / 1000, self._poll_daemon).start()

def _poll_daemon(self):
    """后台线程轮询，用于在 _widget 为 None 时继续尝试"""
    if self._polling_active and self._widget is not None:
        self._widget.after(0, self._poll)  # 切换回主线程轮询
    elif self._polling_active:
        threading.Timer(self._polling_interval_ms / 1000, self._poll_daemon).start()
```

## 8 安全机制

### 8.1 Zip Slip 漏洞防护

`PackageImporter` 在导入 ZIP 项目时检测路径遍历攻击：

```python
def validate_package(self, zip_path):
    with zipfile.ZipFile(zip_path, 'r') as zipf:
        for name in zipf.namelist():
            normalized = os.path.normpath(name)
            if normalized.startswith('..') or os.path.isabs(normalized):
                return False, "ZIP 包含路径遍历条目，可能是恶意文件"
    return True, ""
```

### 8.2 路径遍历防护

`PathResolver` 在解析相对路径时检测路径逃逸：

```python
def to_absolute(self, relative_path: str) -> str:
    abs_path = os.path.normpath(os.path.join(self.project_root, relative_path))
    normalized_root = os.path.normpath(self.project_root)
    if not (abs_path.startswith(normalized_root + os.sep) or abs_path == normalized_root):
        raise ValueError(f"路径逃逸: '{relative_path}' 解析后超出项目根目录")
    return abs_path
```

### 8.3 CodeNode 安全沙箱

`CodeNode` 在执行外部代码前进行 AST 安全检查：

### 10.7 窗口切换例外机制

**SKIP_WINDOW_SWITCH 属性：**

某些节点不需要操作窗口，可以设置 `SKIP_WINDOW_SWITCH = True` 来跳过窗口切换逻辑：

| 节点类型 | SKIP_WINDOW_SWITCH | 说明 |
|---------|-------------------|------|
| DelayNode | True | 延时节点，仅等待时间 |
| SetVariableNode | True | 设变量节点，操作黑板数据（支持常量/变量赋值、递增、删除） |
| AlarmNode | True | 报警节点，播放声音 |
| CodeNode | True | 代码节点，执行外部代码 |
| ScriptNode | True | 脚本节点，执行脚本录制回放 |
| MouseClickNode | False | 需要窗口操作 |
| MouseMoveNode | False | 需要窗口操作 |
| KeyPressNode | False | 需要窗口操作 |
| 其他动作节点 | False | 默认需要窗口操作 |

**实现原理：**

```python
class ActionNode(Node):
    SKIP_WINDOW_SWITCH = False  # 基类默认不跳过

    def _tick_internal(self, context):
        bound_window = context.get_bound_window()
        
        # 只有绑定了窗口且节点不跳过窗口切换时才执行切换逻辑
        if bound_window and not self.SKIP_WINDOW_SWITCH:
            # 窗口切换逻辑...
        else:
            # 直接执行动作
            status = self._execute_action(context)
```

```
输入流程:

    if char == '\n':  → 按下 Enter
    elif char == '\t': → 按下 Tab

## 12 子树引用与多行为树并行架构

### 12.1 功能概述

子树引用机制允许一个行为树节点引用外部行为树项目作为其子树执行，实现行为树的模块化复用和团队协作开发。

多行为树并行功能允许同时运行多个独立的行为树实例，适用于复杂自动化场景的多任务并行处理。

| 能力 | 说明 |
|------|------|
| 子树引用 | SubtreeNode 节点引用外部 .bt 项目文件夹，运行时动态加载 |
| 循环检测 | 加载时 DFS 检测循环引用，拒绝执行循环依赖的树 |
| 黑板隔离 | 支持 inherit/isolated/namespaced 三种黑板隔离模式 |
| 预览展开 | GUI 中可展开查看子树的缩略图结构 |
| 文件监控 | 监控子树文件变化，支持热重载 |
| 多树并行 | MultiTreeManager 管理多个独立 TreeInstance |
| 共享黑板 | 多树之间可选择共享或隔离黑板数据 |

### 12.5.2 多 Tab 隔离机制

| 隔离维度 | 实现方式 | 说明 |
|---------|---------|------|
| 画布 | 每个 Tab 独立 BehaviorTreeCanvas | 使用 place 布局重叠，tkraise() 切换 |
| 引擎 | 每个 Tab 独立 BehaviorTreeEngine | 真正多线程并行执行 |
| 上下文 | 每个 Tab 独立 ExecutionContext | 独立 project_root |
| 黑板 | 每个 Tab 独立 Blackboard | 数据互不干扰 |
| 命令管理 | 每个 Tab 独立 CommandManager | 撤销/重做互不影响 |
| 自动保存 | 每个 Tab 独立 AutoSaveManager | 独立保存到各自文件 |
| 日志 | LogManager 按 Tab 名隔离停止状态 | 停止单个 Tab 不影响其他 Tab 日志 |
| 脚本节点 | ScriptNode 实例级执行器 | 停止单个 Tab 不影响其他 Tab 脚本 |
| 选中状态 | 切换 Tab 时保存/恢复 selected_node_id | 切换回 Tab 时恢复选中 |

### 12.5.3 设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| Tab ID 生成 | 类级别递增计数器 | 关闭后重建不会产生重复 ID |
| 画布布局 | place + tkraise() | 所有画布重叠在同一位置，切换时提升到顶层 |
| 代理属性 | fallback + active_tab | 兼容旧代码中 self.canvas / self.engine 的访问方式 |
| 导入项目 | 直接打开项目路径 | 不复制到工作区，类似浏览器打开标签页 |
| 开始/停止按钮 | 控制所有 Tab | 顶部按钮控制全部，Tab 内按钮控制单个 |
| 全局快捷键 | F10/F12 控制所有 Tab | 继承原有快捷键绑定 |
| 关闭 Tab | 切换到相邻 Tab | 类似浏览器关闭标签页行为 |

### 12.5.4 节点状态路由机制

多 Tab 并行运行时，节点执行状态必须正确路由到对应的画布，避免跨画布状态显示错误：

| 机制 | 说明 |
|------|------|
| `_on_node_status(node_id, status, tab_id)` | 节点状态回调携带 tab_id 参数 |
| 闭包绑定 | `_handle_tab_run` 中通过闭包将 tab_id 绑定到回调 |
| 精确路由 | 根据 tab_id 查找对应 Tab 的画布实例更新状态 |
| 音效跳过 | `start_tab(skip_sound=True)` 避免节点启动/停止时重复播放音效 |

**状态路由流程：**

```
节点执行 → context.notify_node_status()
  → context._on_node_status(node_id, status)
    → Editor._on_node_status(node_id, status, tab_id)
      → instance = tab_manager.get_tab(tab_id)
      → instance.canvas.set_node_status(node_id, node_status)
```

**音效去重机制：**

通过节点启动/停止行为树时，节点自身播放音效，同时传入 `skip_sound=True` 跳过 `_handle_tab_run`/`_handle_tab_stop` 中的默认音效，避免重复播放。

### 12.6 设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 子树路径 | 项目文件夹而非JSON文件 | 包含 project.json + tree.json + images/ + scripts/ 完整资源 |
| 文件夹复制 | 自动复制到 subtrees/ | 确保项目自包含，路径稳定 |
| 循环检测时机 | 加载时检测 | 运行时检测代价高，且可能导致部分执行后失败 |
| 循环检测算法 | DFS + 路径归一化 | O(N) 时间复杂度，覆盖间接循环引用 |
| 黑板默认模式 | inherit | 与原设计兼容，大多数场景不需要隔离 |
| 预览 tag 独立 | 使用 subtree_preview_{id} tag | 避免 canvas.delete(node_id) 误删预览项 |
| 预览坐标 | 屏幕坐标 (_transform_x/y) | 缩放/平移后位置正确 |
| 预览布局 | 从原点开始居中对齐 | 避免双重偏移导致节点右移 |
| 多树线程安全 | threading.Lock | 防止并发操作导致的状态不一致 |
| 文件监控 | mtime 对比 | 轻量级，无需 inotify/filesystem watcher |

### 12.7 项目文件夹中的子树目录

子树引用的项目文件夹结构：

```
MyProject/
├── project.json
├── tree.json
├── subtrees/                    # 子树项目副本存放目录
│   ├── login_flow/             # 登录流程子树
│   │   ├── project.json
│   │   ├── tree.json
│   │   ├── images/
│   │   └── scripts/
│   └── battle_logic/           # 战斗逻辑子树
│       ├── project.json
│       ├── tree.json
│       ├── images/
│       └── scripts/
├── images/
├── scripts/
└── audio/
```

用户选择外部项目文件夹后，FolderField 会自动将整个文件夹复制到当前项目的 `subtrees/` 目录下，并将 `subtree_path` 设置为相对路径 `./subtrees/{folder_name}`。
