## 3 按键输入与画布快捷键冲突分析

### 3.1 按键输入获取方式

#### 3.1.1 KeyField（按键捕获字段）

**文件位置**: [bt_gui/bt_editor/property.py](file:///d:/workspace/autodoor_behavior_tree/bt_gui/bt_editor/property.py#L993-L1092)

```python
class KeyField(FieldWidget):
    def _start_listening(self):
        if self._listening:
            return
        
        self._listening = True
        self.btn.configure(text="请按键...", fg_color=self._dark_colors['warning'])
        
        def on_key_press(event):
            if not self._listening:
                return
            
            # 按键名称映射
            key_mappings = {
                "Control_L": "ctrl", "Control_R": "ctrl",
                "Alt_L": "alt", "Alt_R": "alt",
                "Shift_L": "shift", "Shift_R": "shift",
                # ... 更多映射
            }
            
            key_name = event.keysym
            if key_name in key_mappings:
                key_name = key_mappings[key_name]
            
            self.var.set(key_name)
            self.on_change(self.key, key_name)
            
            # 解绑事件
            toplevel = self.winfo_toplevel()
            toplevel.unbind("<KeyPress>")
            
            return "break"  # ✅ 阻止事件继续传播
        
        # 绑定到顶级窗口
        toplevel = self.winfo_toplevel()
        toplevel.bind("<KeyPress>", on_key_press)
```

**关键机制**:
1. 绑定到顶级窗口的 `<KeyPress>` 事件
2. 捕获按键后立即解绑
3. 返回 `"break"` 阻止事件继续传播

### 3.3 冲突分析

#### 3.3.1 事件传播机制

```
Tkinter 事件传播顺序：

1. 顶级窗口 (BehaviorTreeApp)
   └── bind() 绑定的事件处理器

2. 子组件 (KeyField 等)
   └── bind() 绑定的事件处理器

事件处理流程：
┌─────────────────────────────────────────────────────────────────────┐
│  用户按下按键                                                          │
│      │                                                               │
│      ▼                                                               │
│  ┌──────────────────────────────────────┐                           │
│  │ 顶级窗口事件处理器 (最先触发)          │                           │
│  │ - KeyField 的 <KeyPress> 绑定         │                           │
│  │ - BehaviorTreeApp 的快捷键绑定        │                           │
│  └──────────────────────────────────────┘                           │
│      │                                                               │
│      │ 如果返回 "break"，事件传播终止                                  │
│      │ 如果未返回 "break"，继续传播                                    │
│      ▼                                                               │
│  ┌──────────────────────────────────────┐                           │
│  │ 子组件事件处理器                      │                           │
│  └──────────────────────────────────────┘                           │
└─────────────────────────────────────────────────────────────────────┘
```

#### 3.3.2 KeyField 捕获按键时的行为

```
用户在 KeyField 中捕获按键：

1. 用户点击【修改】按钮
   └── _listening = True
   └── 按钮显示 "请按键..."

2. 用户按下任意键（如 Ctrl+S）
   └── 触发 on_key_press(event)
       ├── 捕获按键名称
       ├── 更新输入框值
       ├── 解绑 <KeyPress> 事件
       └── 返回 "break"  ← 阻止事件传播

3. 结果
   └── KeyField 捕获到按键 "ctrl" 或 "s"
   └── 保存快捷键 Ctrl+S 不会被触发 ✅
```

**结论**: KeyField 捕获按键时，**不会触发画布快捷键**，因为：
1. KeyField 绑定到顶级窗口的 `<KeyPress>` 事件
2. 事件处理器返回 `"break"` 阻止事件继续传播
3. 快捷键绑定（如 `<Control-s>`）不会收到该事件

#### 3.3.3 KeyPressNode 执行按键时的行为

```
KeyPressNode 执行按键：

1. 行为树引擎调用 KeyPressNode._execute_action()
   └── context.execute_key_press(key, action, duration)

2. InputController 执行按键
   └── pyautogui.press(key)  ← 硬件级模拟

3. PyAutoGUI 按键模拟
   ├── 不经过 Tkinter 事件系统
   ├── 直接发送到操作系统
   └── 目标应用程序接收按键

4. 结果
   └── GUI 应用程序不会收到该按键事件 ✅
   └── 画布快捷键不会被触发 ✅
```

**结论**: KeyPressNode 执行按键时，**不会触发画布快捷键**，因为：
1. PyAutoGUI 是硬件级输入模拟
2. 按键事件直接发送到操作系统，不经过 GUI 事件系统
3. 当前应用程序窗口可能不在焦点，不会收到按键事件

### 3.4 潜在冲突场景分析

#### 3.4.1 场景1：KeyField 捕获快捷键组合

**场景描述**: 用户在 KeyField 中按下 Ctrl+S 试图捕获快捷键组合

**分析**:
```
用户操作: 在 KeyField 中按下 Ctrl+S

事件处理:
1. <KeyPress> 事件触发
2. on_key_press() 被调用
3. event.keysym = "s" 或 "Control_L"（取决于按键顺序）
4. 返回 "break"，事件传播终止
5. 快捷键 Ctrl+S 的绑定不会收到事件

结果: 
- KeyField 捕获到单个按键（s 或 ctrl）
- 保存操作不会触发 ✅
- 这是预期行为，用户正在配置按键
```

**结论**: 无冲突，这是预期行为

#### 3.4.2 场景2：行为树执行时触发快捷键

**场景描述**: 行为树执行 KeyPressNode 模拟 Ctrl+S

**分析**:
```
行为树执行: KeyPressNode(key="s", action="press") + Ctrl 修饰键

执行过程:
1. PyAutoGUI 模拟按键
2. 按键事件发送到操作系统
3. 当前焦点窗口（可能是其他应用）接收按键
4. AutoDoor 应用窗口可能不在焦点

结果:
- 如果 AutoDoor 在焦点，可能触发保存操作
- 但这取决于 PyAutoGUI 的实现和操作系统行为
```

**潜在风险**: 如果行为树执行时 AutoDoor 窗口仍在焦点，模拟的按键可能触发快捷键。

**缓解措施**:
1. 用户通常会在目标应用上运行行为树，AutoDoor 窗口不在焦点
2. 可以考虑在行为树运行时临时禁用快捷键（当前未实现）

#### 3.4.3 场景3：Delete/Backspace 在输入框中

**场景描述**: 用户在输入框中按 Delete/Backspace 删除文字

**分析**:
```
用户操作: 在输入框中按 Delete

事件处理:
1. _delete() 方法被调用
2. 检查焦点组件类型
3. 发现是 "CTkEntry" 或 "Text"
4. 直接返回，不执行删除节点操作

结果:
- 输入框正常删除文字 ✅
- 节点不会被删除 ✅
```

**结论**: 焦点保护机制有效，无冲突

### 3.5 冲突分析总结

| 场景 | 是否冲突 | 说明 |
|------|----------|------|
| KeyField 捕获按键 | ❌ 无冲突 | 返回 "break" 阻止事件传播 |
| KeyPressNode 执行按键 | ⚠️ 低风险 | PyAutoGUI 不经过 GUI 事件系统，但焦点窗口可能受影响 |
| Delete/Backspace 在输入框 | ❌ 无冲突 | 焦点保护机制有效 |
| 行为树运行时快捷键 | ⚠️ 低风险 | AutoDoor 在焦点时可能触发快捷键 |

**建议改进**:
1. 在行为树运行时临时禁用全局快捷键
2. 或者在运行时显示提示，提醒用户切换到目标应用

        pass

### 5.2 KeyField 捕获 Delete/Backspace 触发快捷键问题（已修复）

**问题描述**:
在 KeyField 中修改按键时，按下 Delete 或 Backspace 键会触发画布快捷键删除节点，而不是记录按键。

**问题原因**:

在 Tkinter 中，`<KeyPress>` 和 `<Delete>` 是不同的事件类型：
- `<KeyPress>` 是通用按键事件
- `<Delete>` 是特定按键事件（等同于 `<KeyPress-Delete>`）

当用户按下 Delete 键时，两个事件都会**分别触发**：
1. `<KeyPress>` 事件 → KeyField 的处理器被调用
2. `<Delete>` 事件 → BehaviorTreeApp 的快捷键处理器被调用

返回 `"break"` 只会阻止当前事件的传播，不会阻止另一个事件类型的触发。

**事件触发流程**:
```
用户按下 Delete 键：
┌─────────────────────────────────────────────────────────────────────┐
│  事件1: <KeyPress>                                                   │
│      └── KeyField.on_key_press() 被调用                              │
│          ├── 捕获按键 "delete"                                        │
│          └── 返回 "break" ← 阻止 <KeyPress> 事件传播                  │
│                                                                      │
│  事件2: <Delete>（独立事件，不受 <KeyPress> 的 "break" 影响）         │
│      └── BehaviorTreeApp._delete() 被调用                            │
│          └── 删除选中的节点 ❌                                         │
└─────────────────────────────────────────────────────────────────────┘
```

**修复方案**:

KeyField 需要同时绑定 `<Delete>` 和 `<BackSpace>` 事件，并在这些事件中也返回 `"break"` 来阻止快捷键触发。

```python
# 修复后代码
class KeyField(FieldWidget):
    def __init__(self, master, label: str, key: str, on_change: Callable, **kwargs):
        self._listening = False
        self._bound_keys = []  # 记录所有绑定的事件
        super().__init__(master, label, key, on_change, **kwargs)
        self._create_widget()
    
    def _unbind_all_keys(self):
        """解绑所有事件"""
        toplevel = self.winfo_toplevel()
        for key in self._bound_keys:
            try:
                toplevel.unbind(key)
            except Exception:
                pass
        self._bound_keys = []
    
    def _start_listening(self):
        # ... 
        
        toplevel = self.winfo_toplevel()
        
        # 绑定通用按键事件
        toplevel.bind("<KeyPress>", on_key_press)
        self._bound_keys.append("<KeyPress>")
        
        # 绑定特定按键事件（阻止快捷键触发）
        toplevel.bind("<Delete>", lambda e: on_special_key(e, "delete"))
        self._bound_keys.append("<Delete>")
        
        toplevel.bind("<BackSpace>", lambda e: on_special_key(e, "backspace"))
        self._bound_keys.append("<BackSpace>")
```

## 6 结论与建议

### 6.1 多屏显示器兼容性

**结论**: 项目对多屏显示器具有**完全兼容**的支持。

**关键实现**:
1. `ScreenshotManager` 使用 `ImageGrab.grab(all_screens=True)` 支持多屏截图
2. `InputController` 使用 PyAutoGUI，原生支持虚拟桌面坐标
3. 区域/位置选择使用 `screeninfo` 库计算虚拟桌面边界
4. 所有坐标使用虚拟桌面坐标系统，支持负坐标（副显示器）
5. **放大镜**现已修复，使用与 RegionField 相同的多屏支持方案

### 6.2 按键输入与快捷键冲突

**结论**: 大部分场景**无冲突**，已修复发现的问题。

**已修复问题**:
1. ✅ KeyField 捕获 Delete/Backspace 时触发快捷键的问题

**低风险场景**:
1. ⚠️ 行为树运行时，如果 AutoDoor 窗口仍在焦点，模拟的按键可能触发快捷键

**建议改进**:
```python
# 在 BehaviorTreeEditor 中添加
def _start_running(self):
    # 禁用全局快捷键
    self._disable_shortcuts()
    # ... 启动引擎

def _stop_running(self):
    # ... 停止引擎
    # 恢复全局快捷键
    self._enable_shortcuts()

def _disable_shortcuts(self):
    for key in self._shortcuts:
        self.app.unbind(key)

def _enable_shortcuts(self):
    self.app._setup_shortcuts()
```
